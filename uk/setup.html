Меню

Налаштування

Перед роботою з конфігурацією (config/, .env) варто зрозуміти, як застосунок запускається: точка входу HTTP, bootstrap і консоль. У skeleton це три зони: public/, bootstrap/ та bin/.

Загальна схема

HTTP-запит → public/index.php → bootstrap/app.php → провайдери → $app->run()
CLI-команда  → bin/console.php  → bootstrap (paths + app) → провайдери console → Symfony Console

public/index.php

Єдина публічна точка входу для веб-сервера (Apache, Nginx, PHP built-in server). Document root віртуального хоста має вказувати на public/.

<?php declare(strict_types=1);

use Concept\Core\App;

require_once __DIR__ . '/../vendor/autoload.php';

/** @var App $app */
$app = require_once __DIR__ . '/../bootstrap/app.php';

$app->run();
  • autoload — Composer PSR-4 (vendor/autoload.php).
  • bootstrap/app.php — створює App, реєструє провайдери для HTTP.
  • run() — обробка PSR-7 запиту, маршрутизація, відповідь.

У public/ також лежать статичні файли (assets/, vendor/bootstrap після publish) — вони віддаються напряму веб-сервером, без PHP.

Каталог bootstrap/

Код ініціалізації застосунку, спільний для HTTP і CLI (через paths.php).

bootstrap/
├── app.php              # HTTP: App + провайдери app.php
├── paths.php            # мапа логічних шляхів → каталоги проєкту
└── providers/
    ├── app.php          # провайдери для веб-запитів
    └── console.php      # провайдери для консолі

bootstrap/app.php

<?php declare(strict_types=1);

use Concept\Core\App;

$rootPath = dirname(__DIR__);
/** @var array<string, string> $paths */
$paths = include $rootPath . '/bootstrap/paths.php';
$app = App::create($rootPath, $paths);

$providerFiles = [
    $rootPath . '/bootstrap/providers/app.php',
];
$app->registerServiceProviders($providerFiles);

return $app;
  • $rootPath — абсолютний шлях до кореня проєкту (батьківська папка для bootstrap/).
  • paths.php — повертає масив для PathManager (див. нижче).
  • App::create() — контейнер, ранній error handler, реєстрація PathManager.
  • registerServiceProviders() — кожен файл у списку має повертати масив класів провайдерів.

bootstrap/providers/app.php

Набір сервіс-провайдерів ядра для HTTP: конфіг, помилки, події, HTTP, сесія, лог, Twig, валідація, БД, компоненти тощо. Порядок у масиві важливий — залежності реєструються раніше за залежні сервіси.

Щоб додати власний провайдер для вебу — зареєструйте клас у цьому масиві або підключіть окремий файл у $providerFiles у app.php.

bootstrap/providers/console.php

Скорочений набір для CLI: без View/Validation HTTP-стеку, з ConsoleServiceProvider. Використовується з bin/console.php.

bootstrap/paths.php

Центральне місце, де ви задаєте фізичну структуру проєкту. Файл повертає асоціативний масив: логічний ключ → відносний шлях від кореня проєкту. Ядро звертається до каталогів лише через PathManager::get($key), а не через жорстко прошиті рядки.

Структуру папок можна визначити на свій смак: перейменуйте каталоги, рознесіть код по монорепо або перенесіть логи в var/log — достатньо змінити значення в paths.php, якщо ключі залишаються константами PathManager.

Приклад із skeleton:

<?php

use Concept\Core\Components\Path\PathManager;

return [
    PathManager::BOOTSTRAP_DIR => 'bootstrap',
    PathManager::SRC_DIR => 'src',
    PathManager::CONFIG_DIR => 'config',
    PathManager::MIGRATIONS_DIR => 'database/migrations',
    PathManager::SEEDERS_DIR => 'database/seeders',
    PathManager::PUBLIC_DIR => 'public',
    PathManager::STORAGE_DIR => 'storage',
    PathManager::LOGS_DIR => 'storage/logs',
    PathManager::CACHE_DIR => 'storage/cache',
    PathManager::RESOURCES_DIR => 'resources',
    PathManager::VIEWS_DIR => 'resources/views',
    PathManager::ERRORS_FALLBACK_VIEWS_DIR => 'resources/views/errors/fallback',
];

API PathManager

// Абсолютний шлях: /var/www/my-app/config
$pathManager->get(PathManager::CONFIG_DIR);

// З підшляхом: /var/www/my-app/storage/cache/views
$pathManager->get(PathManager::CACHE_DIR, 'views');

// Довільний шлях від кореня проєкту
$pathManager->root('custom/dir');

Другий аргумент get() — підкаталог усередині зареєстрованого шляху (наприклад view.cache_dir у конфігу додається до CACHE_DIR).

Власна структура каталогів

Приклад: логи та кеш у var/, код у app/:

return [
    PathManager::SRC_DIR => 'app',
    PathManager::CONFIG_DIR => 'etc/config',
    PathManager::LOGS_DIR => 'var/log',
    PathManager::CACHE_DIR => 'var/cache',
    PathManager::VIEWS_DIR => 'app/Resources/views',
    // ... інші ключі за потреби
];

Після зміни шляхів:

  • оновіть composer.json → autoload.psr-4, якщо змінили SRC_DIR;
  • переконайтеся, що веб-сервер дивиться на актуальний PUBLIC_DIR;
  • у config/migrations.php та view.php шляхи узгоджені з новою структурою.

Додавати нові ключі в PathManager у ядрі потрібно лише якщо новий тип шляху використовуватиме саме core; для власного коду достатньо $pathManager->root('my/custom/path').

bin/console.php

Точка входу для CLI (міграції, сидери, route:list, view:clear).

<?php declare(strict_types=1);

use Concept\Core\App;
use Symfony\Component\Console\Application as ConsoleApplication;

require __DIR__ . '/../vendor/autoload.php';

$rootPath = dirname(__DIR__);
$paths = include $rootPath . '/bootstrap/paths.php';
$app = App::create($rootPath, $paths);

$providerFiles = [
    $rootPath . '/bootstrap/providers/console.php',
];
$app->registerServiceProviders($providerFiles);

$container = $app->getContainer();
$console = $container->get(ConsoleApplication::class);

$console->run();

Запуск із кореня проєкту (як у composer.json → script console):

php bin/console.php

Відмінності від HTTP:

  • той самий paths.php і App::create();
  • інший список провайдерів — bootstrap/providers/console.php;
  • замість $app->run() — Symfony Console Application з контейнера.

Чеклист нового проєкту

  1. Скопіювати skeleton або створити структуру з public/index.php, bootstrap/, bin/console.php.
  2. Налаштувати bootstrap/paths.php під свої каталоги.
  3. Підключити потрібні провайдери в bootstrap/providers/app.php та console.php.
  4. Налаштувати веб-сервер на public/, створити .env.
  5. Перейти до Конфігурації — config/ та змінні оточення.
