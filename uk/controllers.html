Контролери

Контролер — клас, який обробляє HTTP-запит після middleware і повертає Psr\Http\Message\ResponseInterface. У Concept Core немає обов’язкового базового класу: це звичайний PHP-клас із публічними методами, прив’язаними до маршрутів.

Див. також: маршрутизація, життєвий цикл → контролер, middleware.

Прив’язка до маршруту

Callable маршруту — масив [Клас::class, 'метод']:

use Concept\App\Controllers\IndexController;

$router->get('/', [IndexController::class, 'index'])->setName('home');
$router->get('/projects/{id:number}', [CabinetProjectController::class, 'show'])
    ->setName('cabinet.project.show');

League Route бере контролер із DI-контейнера; залежності конструктора підставляються автоматично (автовайринг). Метод action викликає RouteStrategy.

Де розміщувати

Типові варіанти (шляхи налаштовуються через paths.php):

  • Застосунок — App/Controllers/ для сторінок «ядра» сайту
  • Компонент — Components/{Name}/Controllers/, маршрути в Component/routes.php

Один контролер — одна зона відповідальності (кабінет, адмін, публічна вітрина). Складну логіку виносьте в Services/.

Мінімальний приклад

<?php declare(strict_types=1);

namespace Concept\App\Controllers;

use Concept\Core\Components\View\Contracts\ViewResponseFactoryInterface;
use Psr\Http\Message\ResponseInterface;

final class IndexController
{
    public function __construct(
        private readonly ViewResponseFactoryInterface $viewResponse,
    ) {}

    public function index(): ResponseInterface
    {
        return $this->viewResponse->create('@frontend/index');
    }
}

Шаблон @frontend/index — логічне ім’я Twig (префікс компонента/модуля). Детальніше — у розділі Views.

Автовайринг і конструктор

Під час bootstrap створюється League Container із делегатом ReflectionContainer: якщо клас не зареєстрований явно, контейнер аналізує конструктор і рекурсивно створює залежності за type-hint. 

// Concept\Core\App::__construct()
$this->container->delegate(new ReflectionContainer(cacheResolutions: true));

Для маршруту [UserController::class, 'index'] League Route робить приблизно $container->get(UserController::class) — один екземпляр контролера на запит (shared за замовчуванням для того самого класу в межах одного HTTP-запиту).

Що передавати в конструктор

Оголошуйте залежності явно — через типи параметрів. Контролер «просить» те, що йому потрібно:

Тип залежності Приклад Коментар
Контракти ядра ResponseFactoryInterface, ConfigInterface Реєстрація в service providers Concept Core
Сервіси застосунку / компонента UserAuthService, AuthService Бізнес-логіка; краще залежність від інтерфейсу, якщо є
Моделі, репозиторії UserModel, ProjectModel Допустимо в MVC-стилі; для складних доменів — обгортка-сервіс
Інфраструктура HTTP/сесії FlashBagInterface Для flash-повідомлень після redirect 
public function __construct(
    private readonly ResponseFactoryInterface $response,
    private readonly ViewResponseFactoryInterface $viewResponse,
    private readonly UserAuthService $auth,
    private readonly FlashBagInterface $flashBag,
    private readonly ProjectModel $projectModel,
) {}

Якщо клас не резолвиться (немає binding, циклічна залежність) — помилка контейнера на етапі запиту. Власні реалізації реєструйте в ServiceProvider компонента або застосунку (детальніше — Контейнер сервісів, Service providers).

Не використовуйте контейнер як service locator

У контейнері зареєстровано ContainerInterface, але не інжектуйте його в контролер і не викликайте $this->container->get(...) у методах action. Це приховує реальні залежності, ускладнює тести й рефакторинг.

Погано (service locator)

public function __construct(
    private readonly ContainerInterface $container,
) {}

public function index(): ResponseInterface
{
    $auth = $this->container->get(UserAuthService::class);
    // ...
}

Добре (явні залежності)

public function __construct(
    private readonly UserAuthService $auth,
    private readonly ViewResponseFactoryInterface $viewResponse,
) {}

public function index(): ResponseInterface
{
    $user = $this->auth->user();
    // ...
}

Що краще не передавати в конструктор

  • ContainerInterface, League\Container\Container — антипатерн service locator (див. вище)
  • Router — для URL є UrlGeneratorInterface / redirectByName()
  • Суперглобали ($_GET, $_POST) — лише через FormRequest або PSR-7

Контейнер у App::getContainer() — для bootstrap, CLI і service providers. У контролерах, middleware і сервісах домену покладайтеся на constructor injection конкретних типів.

Відповіді

Кожен action повертає ResponseInterface. Дві основні фабрики:

Інтерфейс Коли використовувати
ViewResponseFactoryInterface HTML: create('@cabinet/index', ['user' => $user])
ResponseFactoryInterface JSON, redirect, back() 
// Redirect за іменем маршруту
return $this->response->redirectByName('cabinet.login');

// JSON (API)
return $this->response->jsonSuccess(['id' => $id]);

// Flash + redirect після POST
$this->flashBag->set('success', 'Збережено');
return $this->response->redirectByName('admin.users');

Повний опис — Відповіді; Twig-шаблони — Views.

Аргументи методу

RouteStrategy заповнює параметри action у такому пріоритеті:

  1. FormRequest — створення з контейнера, валідація до входу в метод
  2. ServerRequestInterface — поточний PSR-7 запит
  3. Параметри URL — ім’я параметра = ім’я в {id:number}, з приведенням типів через Caster
  4. Значення за замовчуванням у сигнатурі PHP
public function show(int $id): ResponseInterface { /* ... */ }

public function update(UpdateUserRequest $request, int $id): ResponseInterface { /* ... */ }

public function handle(ServerRequestInterface $request): ResponseInterface { /* ... */ }

Сервіси (UserModel, ResponseFactoryInterface тощо) — через конструктор (автовайринг), не через ContainerInterface::get(). У параметри методу — лише FormRequest, ServerRequestInterface і сегменти URL.

FormRequest

Type-hint нащадка FormRequest у action — автоматична валідація до входу в метод. Дані: validated() або toDto() (опційно). Повний опис — у розділі Запити та валідація.

Параметри маршруту

Сегмент {id:number} потрапляє в метод як int $id (якщо імена збігаються). Той самий id доступний і як атрибут запиту: $request->getAttribute('id'). 

$router->get('/edit/{id:number}', [UserController::class, 'edit'])
    ->setName('admin.user.edit');

Авторизація

Перевірку доступу зазвичай виносять у middleware на групу маршрутів. У контролері залишають бізнес-перевірки (наприклад, «проєкт належить користувачу»). 

$project = $this->findUserProject($user, $id);
if ($project === null) {
    $this->flashBag->add('error', 'Not found.');
    return $this->response->redirectByName('cabinet.projects');
}

Рекомендовані патерни

  • Тонкий контролер — orchestration: запит → сервіс → відповідь
  • Один action — одна відповідальність — showLogin, login, не «універсальний» метод
  • POST/PUT/DELETE — FormRequest + redirect (PRG), не повторний рендер форми
  • Іменовані маршрути — redirectByName() замість жорстких URL

Closure та invokable

Маршрут може вказувати на closure або клас з __invoke — підтримується League Route. У продакшн-коді переважно використовують класи контролерів для тестованості та DI.
