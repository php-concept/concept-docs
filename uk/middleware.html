Middleware

Middleware — шар між маршрутизацією та контролером: фільтрує запит, додає контекст, перехоплює винятки або змінює відповідь. Concept Core використовує Psr\Http\Server\MiddlewareInterface (PSR-15) і League Route lazyMiddleware().

Див. також: життєвий цикл → middleware, реєстрація на маршрутах.

Контракт PSR-15

<?php declare(strict_types=1);

namespace App\Http\Middlewares;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;

final class ExampleMiddleware implements MiddlewareInterface
{
    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        // До контролера: перевірки, зміна $request

        $response = $handler->handle($request);

        // Після контролера: заголовки, логування

        return $response;
    }
}

Клас реєструється в контейнері; League Route створює екземпляр через DI при lazyMiddleware(MyMiddleware::class).

Де підключати

  • Глобально — на початку routes/web.php (усі маршрути файлу).
  • Група — $router->group(...)->lazyMiddleware(...).
  • Один маршрут — ланцюжок після get() / post().
  • Компонент — у Component/routes.php (наприклад DebugBar у dev).

Порядок реєстрації визначає порядок обгортання: перший доданий middleware — зовнішній шар (виконується першим на вході).

Middleware ядра

Класи в Concept\Core\Http\Middlewares:

Клас Призначення
HandleValidationExceptionMiddleware Ловить ValidationException: JSON 422 або flash + redirect назад
VerifyCsrfTokenMiddleware CSRF для POST, PUT, DELETE
StorePreviousUrlMiddleware Зберігає URL у сесії для back() / safe redirect
ShareViewDataMiddleware Flash, помилки валідації, csrf_token у Twig
ParseJsonBodyMiddleware Парсить JSON body у getParsedBody() (API)
ForceJsonResponseMiddleware Примусовий Accept/Content-Type JSON

Типовий глобальний стек

У routes/web.php часто підключають у такому порядку:

$router->lazyMiddleware(HandleValidationExceptionMiddleware::class);
$router->lazyMiddleware(VerifyCsrfTokenMiddleware::class);
$router->lazyMiddleware(StorePreviousUrlMiddleware::class);
$router->lazyMiddleware(ShareViewDataMiddleware::class);
  1. Зовнішній try/catch валідації
  2. Перевірка CSRF до зміни стану
  3. Фіксація URL перед рендером
  4. Підготовка даних для шаблонів

CSRF

Токен зберігається в сесії (CsrfTokenManager). У формах — поле _csrf_token; для AJAX — заголовки X-CSRF-TOKEN або X-XSRF-TOKEN. 

<input type="hidden" name="_csrf_token" value="{{ csrf_token }}">

У Twig змінна csrf_token з’являється через ShareViewDataMiddleware.

Middleware авторизації (приклад)

У застосунку — власні класи на групах маршрутів:

  • AuthMiddleware — адмін-зона: сесія, роль admin, статус ACTIVE
  • ActiveUserMiddleware — кабінет: авторизація, статуси pending/blocked

При відмові — redirect на login (часто redirectByName('cabinet.login')), без виконання контролера.

$router->group('/cabinet', function (RouteGroup $route) {
  // ...
})->lazyMiddleware(ActiveUserMiddleware::class);

Атрибути запиту

Middleware може передати дані далі через $request->withAttribute():

  • RequestAttribute::VIEW_PAYLOAD — контекст для Twig
  • RequestAttribute::SAFE_BACK_URL — безпечний URL для ResponseFactory::back()

Контролер або ViewResponseFactory читають атрибути з PSR-7 запиту.

Власний middleware

  1. Створіть клас з MiddlewareInterface.
  2. Залежності — через конструктор (контейнер підставить автоматично).
  3. Підключіть lazyMiddleware глобально, на групу або маршрут.
  4. Перевірте порядок відносно CSRF, auth і validation handler.

Middleware має бути «вузьким»: одна відповідальність. Складну бізнес-логіку краще в сервісі або контролері.

Перегляд middleware

php bin/console.php route:list
php bin/console.php route:list -F

Колонка middleware показує глобальні та локальні класи для кожного маршруту.
