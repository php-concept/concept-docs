Маскування даних

Ядро приховує чутливі значення перед записом у логи, debug-панелі та інші місця, де дані можуть потрапити до сторонніх очей. Компонент Masker рекурсивно обробляє масиви, об’єкти та рядки за правилами ключів і regex.

Реєстрація: MaskerServiceProvider у провайдерах HTTP (перед LogServiceProvider, щоб логер отримав маскер).

Архітектура

КласРоль
MaskerInterfaceКонтракт: mask(mixed $data): mixed
DataMaskerОркестратор правил, рекурсія по структурах
RegexMaskerСтандартні regex для рядків і шаблонів ключів
MaskingRuleInterfaceВласне правило (apply, isSensitiveKey)

За замовчуванням MaskerServiceProvider створює DataMasker з одним правилом RegexMasker. Сервіс зареєстрований як shared singleton.

Вбудовані правила

Ключі (регістронезалежні шаблони) замінюються на ***:

  • імена з password, token, _csrf_token, secret
  • api_key, authorization

Рядки проходять через regex:

  • email → ***@***.***
  • номер картки 9999-9999-9999-9999 → зірочки
  • пари на кшталт password=..., token=..., api_key=... → значення замінюється на *****

Інтеграція з логером

LogServiceProvider передає MaskerInterface у Concept\Core\Components\Logger\Logger. Monolog-процесор маскує $record->context перед записом у файл або stderr. 

$logger->info('User updated', [
    'email' => 'user@example.com',
    'password' => 'secret123',  // у лог потрапить ***
]);

Повідомлення ($message) не маскується автоматично — не вставляйте паролі та токени у текст повідомлення, лише в context.

Ручне використання

use Concept\Core\Components\Masker\Contracts\MaskerInterface;

/** @var MaskerInterface $masker */
$masker = $container->get(MaskerInterface::class);

$safe = $masker->mask([
    'login' => 'oleh',
    'password' => 'hunter2',
    'card' => '4111-1111-1111-1111',
]);

Метод mask() для масивів і об’єктів спочатку робить deepClone, щоб не змінювати оригінал. maskRecursive() змінює передану структуру на місці — корисно, коли клонування зайве.

Власні правила

У власному провайдері (після MaskerServiceProvider) можна перевизначити binding:

use Concept\Core\Components\Masker\Contracts\MaskerInterface;
use Concept\Core\Components\Masker\DataMasker;
use Concept\Core\Components\Masker\RegexMasker;

$masker = new DataMasker();
$masker->addRule(new RegexMasker(
    customPatterns: ['/iban=\d+/i' => 'iban=***'],
    customKeyPatterns: ['/.*_pin$/i']
));

$container->add(MaskerInterface::class, fn () => $masker)->setShared(true);

Або реалізуйте MaskingRuleInterface для складнішої логіки (наприклад, часткове приховування телефону).

DebugBar та інші компоненти

Компонент DebugBar інжектує MaskerInterface у колектор запиту — у панелі $_POST, $_SESSION тощо показуються вже замасковані значення. Будь-який власний інструмент діагностики варто будувати так само.

Маскування не замінює шифрування в БД і не прибирає чутливі поля з HTTP-відповідей. Воно знижує ризик витоку через логи та dev-інструменти.
