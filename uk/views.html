Меню

Представлення

HTML у Concept Core рендериться через Twig. Шаблони — файли .twig у каталозі views; у контролері відповідь збирається через ViewResponseFactoryInterface::create() (див. Відповіді → HTML).

Низькорівневий рендер — ViewInterface::render() (сервіси, error handler). Шляхи до views налаштовуються в config/view.php і в config.php компонентів.

Простори імен Twig

Шаблон з контролера вказується як @namespace/шлях/без_розширення. Namespace реєструється в конфігурації: 

// config/view.php
'paths' => [
    'frontend'  => '/resources/views/frontend',
    'dashboard' => '/resources/views/dashboard',
],

// src/Components/UserCabinet/config.php
'paths' => [
    'cabinet' => '/src/Components/UserCabinet/Views',
],

Файл src/Components/UserCabinet/Views/index.twig → @cabinet/index.

Каталог resources/views (ключ PathManager::VIEWS_DIR) також додано як fallback без namespace — рідко потрібен, якщо всі шаблони в namespaced paths.

Рендер з контролера

return $this->viewResponse->create('@frontend/index');

return $this->viewResponse->create('@cabinet/index', [
    'user' => $user,
    'latestProject' => $latestProject,
]);

Другий аргумент — змінні лише для цього запиту. Вони зливаються з глобальним контекстом (errors, old, flashes, csrf_token) з ShareViewDataMiddleware.

Layouts і partials

Типова структура:

resources/views/frontend/
├── layouts/base.twig
├── partials/_navbar.twig
├── partials/_flash.twig
└── index.twig

src/Components/UserCabinet/Views/
├── layouts/cabinet.twig
├── layouts/auth.twig
└── index.twig
{# index.twig #}
{% extends '@frontend/layouts/base.twig' %}

{% block title %}Головна{% endblock %}

{% block body %}
    <h1>Контент</h1>
{% endblock %}
{# Підключення фрагмента #}
{% include '@frontend/partials/_flash.twig' %}

Компонентні сторінки часто наслідують спільний layout іншого namespace (наприклад адмін: @auth-admin/users/list.twig → @dashboard/layouts/base.twig).

URL у шаблонах

Розширення UrlExtension (config/view.php):

<a href="{{ path('architecture') }}">Architecture</a>
<a href="{{ route('cabinet.login') }}">Login</a>
<a href="{{ path('admin.user.edit', { id: user.id }) }}">Edit</a>
<link rel="canonical" href="{{ base_url() }}">
  • path() — відносний шлях за іменем маршруту
  • route() — повний URL
  • base_url() — база застосунку

route_namespace

route_namespace — не те саме, що Twig-namespace шаблону (@cabinet/index). Це логічний контекст інтерфейсу: за поточним URL визначається, в якому «оформленні» працює сторінка (публічний сайт, адмінка, кабінет).

Змінна доступна у всіх шаблонах як глобальна: її додає ContextExtension через ViewContextResolver.

Навіщо

У одному застосунку кілька зон з різними layout і стилями. При 404/500 або в спільних partial потрібно знати, який набір views підключати, без перевірки шляху в кожному Twig-файлі.

route_namespace відповідає: префікс URI → ім’я контексту з конфігурації.

Як визначається

  1. У view.contexts (застосунок + компоненти) задають мапу префікс URI → контекст.
  2. ViewContextResolver бере шлях запиту, перебирає префікси від найдовшого (щоб /admin/users збігся з /admin, а не з /).
  3. ContextExtension::getGlobals() повертає route_namespace у Twig.
// config/view.php
'contexts' => [
    '/' => 'frontend',
],

// src/Components/AuthAdmin/config.php
'contexts' => [
    '/admin' => 'dashboard',
],

// src/Components/UserCabinet/config.php
'contexts' => [
    '/cabinet' => 'cabinet',
    '/login' => 'cabinet',
],

Приклад: сторінки помилок

У skeleton один диспетчер у resources/views/errors/ підключає 404/500 відповідного контексту: 

{# resources/views/errors/404.twig #}
{% include '@' ~ (route_namespace|default('frontend')) ~ '/errors/404.twig' %}
  • /unknown-page → frontend → @frontend/errors/404.twig (публічний layout)
  • /admin/missing → dashboard → @dashboard/errors/404.twig (layout адмінки)

Публічна 404 наслідує @frontend/layouts/base.twig; адмінська — @dashboard/layouts/error.twig. Без route_namespace довелося б дублювати умови «якщо /admin — один шаблон, інакше інший».

Інше використання в Twig

{# Різний partial залежно від зони #}
{% if route_namespace == 'dashboard' %}
    {% include '@dashboard/partials/_sidebar.twig' %}
{% else %}
    {% include '@frontend/partials/_navbar.twig' %}
{% endif %}

{# Динамічний include (ignore missing — якщо partial відсутній) #}
{% include '@' ~ route_namespace ~ '/partials/_breadcrumbs.twig' ignore missing %}

Важливий нюанс

Значення в contexts (dashboard, frontend, cabinet) має відповідати Twig-namespace, де реально лежать потрібні шаблони.

У AuthAdmin контекст названо dashboard, а не auth-admin: сторінки помилок і спільний layout адмінки — у @dashboard/, а контент CRUD — у @auth-admin/.

Для /cabinet контекст cabinet, але окремих Views/errors/ у компоненті UserCabinet немає. Якщо потрібна своя 404 у кабінеті — додайте @cabinet/errors/404.twig або змініть диспетчер із fallback на frontend.

Конфігурація config/view.php

return [
    'view' => [
        'extensions' => [
            AppExtension::class,      // has_component()
            ContextExtension::class,  // route_namespace
            UrlExtension::class,      // path(), route(), base_url()
        ],
        'paths' => [ /* namespace => шлях від кореня проєкту */ ],
        'contexts' => [
            '/' => 'frontend',       // URI prefix => ім'я контексту для route_namespace
        ],
        'cache_dir' => 'views',      // storage/cache/views у production
    ],
];
  • extensions — класи Twig\Extension\AbstractExtension, резолвяться з DI-контейнера
  • paths — Twig namespaces для застосунку
  • contexts — префікс URI → route_namespace (детально)
  • cache_dir — підкаталог у storage/cache; при app.debug кеш Twig вимкнено

Views у компонентах

У Component/config.php секція view: paths, contexts, опційно extensions. Під час ComponentsServiceProvider::boot() дані додаються до реєстрів і підхоплюються Twig при HTTP-запиті (не в CLI). 

// AuthAdmin — власне розширення + шаблони
'view' => [
    'extensions' => [ TwigExtension::class ],  // auth()
    'paths' => [
        'auth-admin' => '/src/Components/AuthAdmin/Views',
    ],
    'contexts' => [
        '/admin' => 'dashboard',
    ],
],

У Twig: {{ auth().check() }} (якщо підключено розширення компонента), {{ has_component('UserCabinet') }} — з AppExtension.

Власне Twig-розширення

class MyExtension extends AbstractExtension
{
    public function getFunctions(): array
    {
        return [
            new TwigFunction('format_money', [$this, 'formatMoney']),
        ];
    }

    public function getFilters(): array
    {
        return [
            new TwigFilter('excerpt', [$this, 'excerpt']),
        ];
    }
}

Зареєструйте клас у view.extensions (застосунок) або view.extensions компонента. У шаблоні: {{ price|format_money }}, {{ format_money(price) }}.

Прямий виклик ViewInterface

/** @var ViewInterface $view */
$view = $container->get(ViewInterface::class);
$html = $view->render('@frontend/errors/404', ['exception' => $e]);

Розширення .twig додається автоматично, якщо не вказано. Події: TemplateRendering, TemplateRendered (телеметрія). У debug — DebugExtension і Twig Profiler.

Сторінки помилок

Диспетчер resources/views/errors/{код}.twig через route_namespace підключає @frontend/errors/... або @dashboard/errors/.... У production error handler рендерить той самий шлях; якщо файлу немає — fallback на 500.

Додаткові fallback-файли PHP — resources/views/errors/fallback/ (PathManager::ERRORS_FALLBACK_VIEWS_DIR).

Статика в шаблонах

CSS/JS зазвичай лежать у public/ і підключаються абсолютними шляхами (/assets/css/frontend.css, /vendor/bootstrap/...). Компоненти можуть публікувати assets через assets у config.php (копіювання в public/components/...).

Логіку (запити до БД, права доступу) тримайте в контролерах і сервісах. У Twig — відображення, умови, цикли та допоміжні фільтри.
