Провайдери сервісів

Service provider — клас, який реєструє сервіси в DI-контейнері під час bootstrap. У Concept Core це основний спосіб підключити конфіг, HTTP, БД, Twig, валідацію та модулі-компоненти.

Реалізація — League Container Service Providers (League\Container\ServiceProvider\AbstractServiceProvider). Див. також: Контейнер сервісів, конфігурація.

Анатомія провайдера

<?php declare(strict_types=1);

namespace Concept\App\Providers;

use Concept\Core\Providers\Concerns\PeeksEventDispatcher;
use League\Container\ServiceProvider\AbstractServiceProvider;
use League\Container\ServiceProvider\BootableServiceProviderInterface;

final class BillingServiceProvider extends AbstractServiceProvider implements BootableServiceProviderInterface
{
    use PeeksEventDispatcher;

    public function provides(string $id): bool
    {
        return $id === BillingGatewayInterface::class;
    }

    public function register(): void
    {
        $container = $this->getContainer();

        $container->add(BillingGatewayInterface::class, function () {
            return new StripeBillingGateway(/* ... */);
        })->setShared(true);
    }

    public function boot(): void
    {
        // Eager-логіка одразу після register() цього провайдера
    }
}

register() і boot()

  • register() — $container->add(), bindings, фабрики
  • boot() — додаткова логіка одразу після register() цього ж провайдера (eager), а не після всіх провайдерів у файлі

boot() має сенс лише в класі з BootableServiceProviderInterface. З документації League Container:

In much the same way, this method has access to the container itself and can interact with it however you wish, the difference is that the boot method is invoked as soon as you register the service provider with the container meaning that everything in this method is eagerly loaded.

If you wish to apply inflectors or register further service providers from this one, it must be from a bootable service provider like this one, otherwise they will be ignored.

League Container — Service Providers

Порядок у bootstrap/providers/app.php критичний: для кожного провайдера спочатку register(), потім одразу boot(), і лише потім наступний провайдер у списку. У boot() провайдера N доступні сервіси лише від провайдерів вище у файлі; провайдери нижче ще не виконували register().

Тому ComponentsServiceProvider стоїть останнім: у його boot() вже є ConfigInterface, Router, View registries тощо — і він може addServiceProvider() для провайдерів компонентів, підключити routes.php і Twig paths.

provides()

Список id, які надає провайдер. Дозволяє відкладене завантаження (deferred): провайдер підтягується, коли вперше потрібен один із перелічених сервісів.

Події ServiceAwakening

Провайдери core використовують trait PeeksEventDispatcher: якщо EventDispatcherInterface вже в контейнері, при створенні сервісу диспатчиться ServiceAwakening (телеметрія в debug).

Підключення провайдерів

App::registerServiceProviders() підключає масив класів із PHP-файлу:

foreach ($providers as $providerClassName) {
    $provider = new $providerClassName();
    $this->container->addServiceProvider($provider);
}

Провайдери не реєструються в autowiring — їх інстансують явно (new $providerClassName()).

HTTP — bootstrap/providers/app.php

return [
    ConfigServiceProvider::class,
    ErrorHandlerServiceProvider::class,
    EventServiceProvider::class,
    HttpServiceProvider::class,
    SessionServiceProvider::class,
    MaskerServiceProvider::class,
    LogServiceProvider::class,
    ViewServiceProvider::class,
    ValidationServiceProvider::class,
    DatabaseServiceProvider::class,
    CastingServiceProvider::class,
    DebugLoggerServiceProvider::class,
    ComponentsServiceProvider::class,
];

Порядок важливий: config → events → HTTP/view/DB → components останнім.

CLI — bootstrap/providers/console.php

bin/console.php викликає App::create() і окремий список провайдерів (без View/Validation/Casting за потреби, але з ConsoleServiceProvider): 

// bin/console.php
$app->registerServiceProviders([$rootPath . '/bootstrap/providers/console.php']);

$console = $container->get(ConsoleApplication::class);
$console->run();

ConsoleServiceProvider читає config/commands.php і додає команди в Symfony Console. Команди компонентів — через ComponentsServiceProvider::boot().

Власний провайдер застосунку

  1. Створіть клас у App/Providers/MyServiceProvider.php.
  2. У register() — bindings для інтерфейсів застосунку.
  3. Додайте клас у bootstrap/providers/app.php після провайдерів, від яких залежите.
// bootstrap/providers/app.php
return [
    ConfigServiceProvider::class,
    // ...
    BillingServiceProvider::class,  // після ConfigServiceProvider
    ComponentsServiceProvider::class,
];

Якщо провайдеру потрібен ConfigInterface — ставте його після ConfigServiceProvider. Якщо потрібен Router — після HttpServiceProvider.

Провайдери компонентів

Компонент оголошує провайдери в Component/config.php. ComponentsServiceProvider::boot() підключає їх динамічно: 

// src/Components/DebugBar/config.php
'providers' => [
    DebugBarServiceProvider::class,
    MaskerServiceProvider::class,  // якщо ще не в app.php
],
// Concept\Core\Providers\ComponentsServiceProvider
foreach ($registry->providers() as $providerClass) {
    $this->getContainer()->addServiceProvider(new $providerClass());
}

Список активних компонентів — config/components.php (у dev може бути config/dev/components.php з DebugBar). Клас компонента (AuthAdminComponent) читає свій config.php.

Що робить ComponentsServiceProvider::boot()

  1. Події ComponentRegistering для кожного модуля
  2. Реєстрація console-команд компонентів
  3. Шляхи міграцій і seeders у registries
  4. Nested service providers компонентів
  5. require файлів Component/routes.php (змінна $router)
  6. У HTTP (не CLI) — Twig paths, extensions, view contexts з компонентів

Тому маршрути компонента з’являться лише якщо компонент увімкнено в config і ComponentsServiceProvider виконав boot().

Приклади з core

HttpServiceProvider — register + boot

register(): PSR-7, Router + RouteStrategy + завантаження routes/*.php.

boot(): резолвери сторінок для Illuminate Paginator.

League Route: документація маршрутизації.

DatabaseServiceProvider — чому boot

У register() — DatabaseInterface, Migrator, registries. У boot() (одразу після register() цього провайдера) — створення CapsuleManager, підключення з ConfigInterface (Config уже зареєстрований вище в списку), listeners SQL. Capsule додається в контейнер уже налаштованим.

ViewServiceProvider — Twig

Збирає Environment Twig: cache, debug, extensions з config, paths з registries. Після boot компонентів у registries з’являються додаткові namespaces.

Twig: twig.symfony.com, views — Представлення.

DebugBarServiceProvider (компонент)

Реєструє CustomDebugBar і JavascriptRenderer з колекторами телеметрії. Підключається лише якщо DebugBarComponent у dev-config компонентів.

Рекомендації

  • Один провайдер — одна область (Billing, Notifications), не «GodProvider»
  • register — лише bindings; важкі підключення — у boot()
  • Залежності через $container->get() лише всередині провайдера, не в контролерах
  • setShared(true) для сервісів, що мають бути singleton на запит
  • Не змінюйте порядок core-провайдерів без розуміння залежностей

Чеклист нового модуля

  1. Компонент у config/components.php
  2. За потреби — providers у Component/config.php
  3. routes.php, view.paths, migrations — у config компонента
  4. Класи сервісів резолвляться через autowiring або binding у провайдері модуля

Якщо сервіс «не знаходиться» в контейнері — перевірте, чи провайдер у правильному файлі (app.php vs console.php), чи виконався boot() компонента, і чи не потрібен явний add() замість autowiring.
