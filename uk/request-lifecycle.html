Меню

Життєвий цикл запиту

HTTP-запит проходить від веб-сервера через public/index.php, bootstrap провайдерів і диспетчеризацію League Route. Нижче — повний ланцюжок з урахуванням middleware, компонентів і обробки помилок.

Передумови: налаштування точок входу, конфігурація.

Огляд

Веб-сервер → public/index.php
    → vendor/autoload.php
    → bootstrap/app.php (App::create, провайдери)
    → App::run()
        → Router::dispatch(PSR-7 Request)
            → глобальний middleware (CSRF, сесія, views…)
            → middleware маршруту (якщо є)
            → RouteStrategy → контролер / closure
        → SapiEmitter → відповідь клієнту

1. Точка входу

public/index.php — єдиний файл, на який має дивитися document root. Він не містить бізнес-логіки: 

require_once __DIR__ . '/../vendor/autoload.php';
$app = require_once __DIR__ . '/../bootstrap/app.php';
$app->run();

2. Bootstrap (до run)

bootstrap/app.php виконує:

  1. App::create($rootPath, $paths) — контейнер League, PathManager, ранній Whoops (з APP_DEBUG з оточення).
  2. registerServiceProviders() — підключення списку з bootstrap/providers/app.php.

Кожен провайдер реєструє сервіси в контейнері; провайдери з BootableServiceProviderInterface також викликають boot().

Порядок провайдерів (HTTP)

Типовий порядок у bootstrap/providers/app.php:

  1. Config — конфіг, .env, часовий пояс
  2. ErrorHandler — Whoops (HTML / JSON / production fallback)
  3. Event — диспетчер подій
  4. Http — PSR-7 запит, Router, фабрики відповідей
  5. Session — Symfony Session (стартує при першому зверненні)
  6. Log, View, Validation, Database, Caster
  7. Components — реєстрація модулів (див. крок 3)

Що робить ComponentsServiceProvider::boot()

Під час boot() (ще до run()), якщо увімкнені компоненти:

  • реєструються консольні команди, міграції, сидери компонентів;
  • підключаються додаткові service providers компонентів;
  • require файлів routes.php кожного компонента — маршрути додаються в той самий Router;
  • реєструються Twig paths, extensions, contexts для views компонентів.

Глобальні маршрути з routes/web.php підключаються раніше — у HttpServiceProvider з масиву config/routes.php.

3. PSR-7 запит

ServerRequestInterface створюється в контейнері через Laminas\Diactoros\ServerRequestFactory::fromGlobals() — дані з $_SERVER, $_GET, $_POST, $_COOKIE, $_FILES.

Після збігу маршруту RouteStrategy додає параметри URL у атрибути запиту ({id} → $request->getAttribute('id')) і оновлює екземпляр у контейнері.

4. App::run()

$router = $this->container->get(Router::class);
$request = $this->container->get(ServerRequestInterface::class);
$response = $router->dispatch($request);
(new SapiEmitter)->emit($response);

Router налаштований із RouteStrategy — кастомна стратегія Concept Core замість стандартної League: DI параметрів контролера, FormRequest, події маршруту.

5. Middleware

Глобальний middleware реєструється в файлах маршрутів через $router->lazyMiddleware(ClassName::class) (League Route). Middleware створюється з контейнера при обробці запиту.

Приклад із routes/web.php:

  • HandleValidationExceptionMiddleware — перетворює помилки валідації на відповідь
  • VerifyCsrfTokenMiddleware — CSRF для POST/PUT/DELETE
  • StorePreviousUrlMiddleware — зберігає URL для редіректів
  • ShareViewDataMiddleware — спільні змінні для Twig

Кожен middleware реалізує MiddlewareInterface::process($request, $handler) і викликає $handler->handle($request) для передачі далі по ланцюгу. Детальніше — у розділі Middleware.

6. Контролер і RouteStrategy

Після middleware виконується callable маршруту, наприклад:

$router->get('/', [IndexController::class, 'index'])->setName('home');

RouteStrategy розв’язує аргументи методу action:

  1. FormRequest — створення з контейнера, валідація, події FormRequestValidating / FormRequestValidated
  2. ServerRequestInterface — поточний запит
  3. Параметри URL — з маршруту, з приведенням типів через Caster
  4. Значення за замовчуванням — якщо оголошені в сигнатурі

Залежності сервісів (ViewResponseFactoryInterface, моделі тощо) — через конструктор контролера (автовайринг League Container + ReflectionContainer). Детальніше — Контролери → автовайринг, аргументи методу.

Події RouteCallableInvoking / RouteCallableInvoked дозволяють вимірювати час виконання handler.

7. Відповідь

Контролер повертає ResponseInterface (PSR-7). Типові варіанти:

  • ViewResponseFactoryInterface::create('@frontend/index') — HTML через Twig
  • ResponseFactoryInterface — JSON, redirect, порожня відповідь

SapiEmitter (Laminas) відправляє заголовки та тіло в PHP SAPI (Apache, FPM, built-in server). Детальніше — Відповіді.

8. Обробка помилок

  • До bootstrap — ранній Whoops у App::create() (читає APP_DEBUG з $_ENV).
  • Після Config — ErrorHandlerServiceProvider::boot() перемикає Whoops: PrettyPage (debug), JSON (API), production fallback views.
  • Валідація — ValidationException з FormRequest; middleware може перехопити й повернути 422 / redirect.
  • CSRF — CsrfException при невалідному токені.
  • 404 / метод — League Route; обробка залежить від налаштувань router.

9. Події (опційно)

Якщо app.debug увімкнено, EventServiceProvider буферизує події framework/HTTP для debug-панелі. Підписники можна додавати на події ядра та власні події застосунку.

CLI — окремий цикл

bin/console.php не викликає App::run(). Використовує той самий paths.php, але провайдери з bootstrap/providers/console.php і Symfony Console Application::run(). HTTP middleware і PSR-7 запит у CLI не застосовуються.
