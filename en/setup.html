Setup

Before working with configuration (config/, .env), it helps to understand how the application starts: the HTTP entry point, bootstrap, and console. These are three areas: public/, bootstrap/, and bin/.

Overview

HTTP request → public/index.php → bootstrap/app.php → providers → $app->run()
CLI command  → bin/console.php  → bootstrap (paths + app) → console providers → Symfony Console

public/index.php

The single public entry point for the web server (Apache, Nginx, PHP built-in server). The virtual host document root must point to public/.

<?php declare(strict_types=1);

use Concept\Core\App;

require_once __DIR__ . '/../vendor/autoload.php';

/** @var App $app */
$app = require_once __DIR__ . '/../bootstrap/app.php';

$app->run();
  • autoload — Composer PSR-4 (vendor/autoload.php).
  • bootstrap/app.php — creates App, registers providers for HTTP.
  • run() — PSR-7 request handling, routing, response.

The public/ directory also holds static files (assets/, vendor/bootstrap after publish) — they are served directly by the web server, without PHP.

bootstrap/ directory

Application initialization code, shared by HTTP and CLI (via paths.php).

bootstrap/
├── app.php              # HTTP: App + app.php providers
├── paths.php            # map of logical paths → project directories
└── providers/
    ├── app.php          # providers for web requests
    └── console.php      # providers for console

bootstrap/app.php

<?php declare(strict_types=1);

use Concept\Core\App;

$rootPath = dirname(__DIR__);
/** @var array<string, string> $paths */
$paths = include $rootPath . '/bootstrap/paths.php';
$app = App::create($rootPath, $paths);

$providerFiles = [
    $rootPath . '/bootstrap/providers/app.php',
];
$app->registerServiceProviders($providerFiles);

return $app;
  • $rootPath — absolute path to the project root (parent directory of bootstrap/).
  • paths.php — returns an array for PathManager (see below).
  • App::create() — container, early error handler, PathManager registration.
  • registerServiceProviders() — each file in the list must return an array of provider classes.

bootstrap/providers/app.php

The core service provider set for HTTP: config, errors, events, HTTP, session, log, Twig, validation, database, components, and more. Order in the array matters — dependencies are registered before dependent services. Detailed description of each core provider — what it is for and what it registers in the container.

To add your own provider for the web — register the class in this array or include a separate file in $providerFiles in app.php.

bootstrap/providers/console.php

A reduced set for CLI: without the View/Validation HTTP stack, with ConsoleServiceProvider. Used from bin/console.php.

bootstrap/paths.php

The central place where you define the physical project structure. The file returns an associative array: logical key → relative path from the project root. The core accesses directories only through PathManager::get($key), not hard-coded strings.

You can define the folder structure to your liking: rename directories, split code across a monorepo, or move logs to var/log — just change the values in paths.php, as long as the keys remain PathManager constants.

Example:

<?php

use Concept\Core\Components\Path\PathManager;

return [
    PathManager::BOOTSTRAP_DIR => 'bootstrap',
    PathManager::SRC_DIR => 'src',
    PathManager::CONFIG_DIR => 'config',
    PathManager::MIGRATIONS_DIR => 'database/migrations',
    PathManager::SEEDERS_DIR => 'database/seeders',
    PathManager::PUBLIC_DIR => 'public',
    PathManager::STORAGE_DIR => 'storage',
    PathManager::LOGS_DIR => 'storage/logs',
    PathManager::CACHE_DIR => 'storage/cache',
    PathManager::RESOURCES_DIR => 'resources',
    PathManager::VIEWS_DIR => 'resources/views',
    PathManager::ERRORS_FALLBACK_VIEWS_DIR => 'resources/views/errors/fallback',
];

PathManager keys

Constant (array key) Path Used by
BOOTSTRAP_DIRbootstrapReserved for bootstrap resources
SRC_DIRsrcApplication code (PSR-4)
CONFIG_DIRconfigPHP config, config/{env}/ overrides
MIGRATIONS_DIRdatabase/migrationsDatabase migrations
SEEDERS_DIRdatabase/seedersSeeders
PUBLIC_DIRpublicDocument root, static assets
STORAGE_DIRstorageApplication writable data
LOGS_DIRstorage/logsMonolog, debug log
CACHE_DIRstorage/cacheTwig cache, Valinor, etc.
RESOURCES_DIRresourcesResources, not only views
VIEWS_DIRresources/viewsTwig templates (base path)
ERRORS_FALLBACK_VIEWS_DIRresources/views/errors/fallbackError fallback templates (production)

PathManager API

// Absolute path: /var/www/my-app/config
$pathManager->get(PathManager::CONFIG_DIR);

// With subpath: /var/www/my-app/storage/cache/views
$pathManager->get(PathManager::CACHE_DIR, 'views');

// Arbitrary path from project root
$pathManager->root('custom/dir');

The second argument to get() is a subdirectory within the registered path (for example view.cache_dir in config is appended to CACHE_DIR).

Custom directory structure

Example: logs and cache in var/, code in app/:

return [
    PathManager::SRC_DIR => 'app',
    PathManager::CONFIG_DIR => 'etc/config',
    PathManager::LOGS_DIR => 'var/log',
    PathManager::CACHE_DIR => 'var/cache',
    PathManager::VIEWS_DIR => 'app/Resources/views',
    // ... other keys as needed
];

After changing paths:

  • update composer.json → autoload.psr-4 if you changed SRC_DIR;
  • ensure the web server points to the current PUBLIC_DIR;
  • in config/migrations.php and view.php, paths align with the new structure.

Adding new keys to PathManager in the core is only needed if a new path type will be used by the core itself; for your own code, $pathManager->root('my/custom/path') is enough.

bin/console.php

CLI entry point (migrations, seeders, route:list, view:clear).

<?php declare(strict_types=1);

use Concept\Core\App;
use Symfony\Component\Console\Application as ConsoleApplication;

require __DIR__ . '/../vendor/autoload.php';

$rootPath = dirname(__DIR__);
$paths = include $rootPath . '/bootstrap/paths.php';
$app = App::create($rootPath, $paths);

$providerFiles = [
    $rootPath . '/bootstrap/providers/console.php',
];
$app->registerServiceProviders($providerFiles);

$container = $app->getContainer();
$console = $container->get(ConsoleApplication::class);

$console->run();

Run from the project root (as in composer.json → script console):

php bin/console.php

Differences from HTTP:

  • the same paths.php and App::create();
  • a different provider list — bootstrap/providers/console.php;
  • instead of $app->run() — Symfony Console Application from the container.

New project checklist

  1. Copy the skeleton or create a structure with public/index.php, bootstrap/, bin/console.php.
  2. Configure bootstrap/paths.php for your directories.
  3. Register the providers you need in bootstrap/providers/app.php and console.php.
  4. Point the web server at public/, create .env.
  5. Continue to Configuration — config/ and environment variables.
