Request Lifecycle

An HTTP request flows from the web server through public/index.php, provider bootstrap, and League Route dispatch. Below is the full chain including middleware, components, and error handling.

Prerequisites: setup entry points, configuration.

Overview

Web server → public/index.php
    → vendor/autoload.php
    → bootstrap/app.php (App::create, providers)
    → App::run()
        → Router::dispatch(PSR-7 Request)
            → global middleware (CSRF, session, views…)
            → route middleware (if any)
            → RouteStrategy → controller / closure
        → SapiEmitter → response to client

1. Entry point

public/index.php is the only file the document root should point to. It contains no business logic: 

require_once __DIR__ . '/../vendor/autoload.php';
$app = require_once __DIR__ . '/../bootstrap/app.php';
$app->run();

2. Bootstrap (before run)

bootstrap/app.php performs:

  1. App::create($rootPath, $paths) — League container, PathManager, early Whoops (from APP_DEBUG in the environment).
  2. registerServiceProviders() — loads the list from bootstrap/providers/app.php.

For each provider in the list: register(), then immediately boot() (if BootableServiceProviderInterface is implemented) — see Service providers → register and boot.

Provider order (HTTP)

Typical order in bootstrap/providers/app.php:

  1. Config — config, .env, timezone
  2. ErrorHandler — Whoops (HTML / JSON / production fallback)
  3. Event — event dispatcher
  4. Http — PSR-7 request, Router, response factories
  5. Session — Symfony Session (starts on first access)
  6. Log, View, Validation, Database, Caster
  7. Components — module registration (see step 3)

What ComponentsServiceProvider::boot() does

During boot(), ComponentsServiceProvider (right after its registration, still before run()), if components are enabled:

  • registers component console commands, migrations, and seeders;
  • loads additional component service providers;
  • requires each component’s routes.php — routes are added to the same Router;
  • registers Twig paths, extensions, and contexts for component views.

Global routes from routes/web.php are loaded earlier — in HttpServiceProvider from the config/routes.php array.

3. PSR-7 request

ServerRequestInterface is created in the container via Laminas\Diactoros\ServerRequestFactory::fromGlobals() — data from $_SERVER, $_GET, $_POST, $_COOKIE, $_FILES.

After a route match, RouteStrategy adds URL parameters to request attributes ({id} → $request->getAttribute('id')) and updates the instance in the container.

4. App::run()

$router = $this->container->get(Router::class);
$request = $this->container->get(ServerRequestInterface::class);
$response = $router->dispatch($request);
(new SapiEmitter)->emit($response);

Router is configured with RouteStrategy — Concept Core’s custom strategy instead of the default League one: DI for controller parameters, FormRequest, route events.

5. Middleware

Global middleware is registered in route files via $router->lazyMiddleware(ClassName::class) (League Route). Middleware is created from the container when the request is handled.

Example from routes/web.php:

  • HandleValidationExceptionMiddleware — turns validation errors into a response
  • VerifyCsrfTokenMiddleware — CSRF for POST/PUT/DELETE
  • StorePreviousUrlMiddleware — stores URL for redirects
  • ShareViewDataMiddleware — shared variables for Twig

Each middleware implements MiddlewareInterface::process($request, $handler) and calls $handler->handle($request) to pass control down the chain. More details in the Middleware section.

6. Controller and RouteStrategy

After middleware, the route callable runs, for example:

$router->get('/', [IndexController::class, 'index'])->setName('home');

RouteStrategy resolves action method arguments:

  1. FormRequest — created from the container, validated, events FormRequestValidating / FormRequestValidated
  2. ServerRequestInterface — current request
  3. URL parameters — from the route, with type casting via Caster
  4. Default values — if declared in the signature

Service dependencies (ViewResponseFactoryInterface, models, and so on) — via the controller constructor (League Container autowiring + ReflectionContainer). More details — Controllers → autowiring, method arguments.

Events RouteCallableInvoking / RouteCallableInvoked allow measuring handler execution time.

7. Response

The controller returns ResponseInterface (PSR-7). Typical options:

  • ViewResponseFactoryInterface::create('@frontend/index') — HTML via Twig
  • ResponseFactoryInterface — JSON, redirect, empty response

SapiEmitter (Laminas) sends headers and body to the PHP SAPI (Apache, FPM, built-in server). More details — Responses.

8. Error handling

  • Before bootstrap — early Whoops in App::create() (reads APP_DEBUG from $_ENV).
  • After Config — ErrorHandlerServiceProvider::boot() switches Whoops: PrettyPage (debug), JSON (API), production fallback views.
  • Validation — ValidationException from FormRequest; middleware can catch it and return 422 / redirect.
  • CSRF — CsrfException on invalid token.
  • 404 / method — League Route; handling depends on router settings.

9. Events (optional)

When app.debug is enabled, EventServiceProvider buffers framework/HTTP events for the debug panel. Subscribers can be added for core events and custom application events.

CLI — separate lifecycle

bin/console.php does not call App::run(). It uses the same paths.php, but providers from bootstrap/providers/console.php and Symfony Console Application::run(). HTTP middleware and the PSR-7 request do not apply in CLI.
