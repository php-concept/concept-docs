Database

Concept Core uses Illuminate Database (Capsule + Eloquent) — the same layer as Laravel Eloquent. Connections, models, migrations, and seeders are integrated through the core DatabaseServiceProvider.

HTTP and CLI share one configuration (config/db.php, .env); the schema is updated with console db:* commands (see Console).

Connection

Settings from config/db.php are overridden by environment variables (DB_HOST → db.host, and so on). For the full table, see Configuration → Database variables. 

// config/db.php
return [
    'db' => [
        'driver'   => 'mysql',
        'host'     => '127.0.0.1',
        'port'     => '3306',
        'database' => 'concept_db',
        'username' => 'root',
        'password' => '',
        'charset'  => 'utf8mb4',
    ],
];

In DatabaseServiceProvider::boot(), a global Illuminate\Database\Capsule\Manager is created, bootEloquent() is called, and an SQL listener is registered (logged when app.debug or log.query is enabled).

The low-level facade is DatabaseInterface (a wrapper around Capsule) when you need access outside Eloquent models.

Eloquent models

The application extends Concept\App\Models\BaseModel (which extends Illuminate\Database\Eloquent\Model). Components declare models in their own namespace, for example Concept\Components\AuthAdmin\Models\UserModel.

Recommended practices:

  • FIELD_* field constants instead of “magic” strings in queries and fillable;
  • the IdTrait trait for typed getId();
  • $fillable, $hidden, $casts — same as in Laravel;
  • inject the model into a controller or service constructor via the container (autowiring).
class UserModel extends BaseModel
{
    use IdTrait;

    public const string FIELD_EMAIL = 'email';

    protected $table = 'users';

    protected $fillable = [self::FIELD_EMAIL, /* ... */];
}

Queries in code

Build queries through $model->newQuery() (query builder / Eloquent), not via static calls on the model class. This aligns with injecting the model in the constructor. 

public function __construct(
    private readonly UserModel $userModel,
    private readonly ConfigInterface $config,
) {}

public function index(): ResponseInterface
{
    $users = $this->userModel
        ->newQuery()
        ->orderBy(UserModel::FIELD_CREATED_AT, BaseModel::SORT_DESC)
        ->paginate($this->config->getInt('pagination.per_page', 10))
        ->withQueryString();

    return $this->viewResponse->create('@auth-admin/users/list', ['users' => $users]);
}

Create records after validation with newQuery()->create([...]) or validated() from FormRequest (see Requests and validation). Official references: Eloquent, Query Builder, Pagination.

Migrations

Migration directories are registered in config/migrations.php (application) and in the component config.php (migrations.paths section). During ComponentsServiceProvider::boot(), paths from all enabled components are added to MigrationRegistry. 

// config/migrations.php
return [
    'migrations' => [
        'table' => 'migrations',
        'paths' => ['database/migrations'],
    ],
];

// src/Components/AuthAdmin/config.php
'migrations' => [
    'paths' => ['src/Components/AuthAdmin/Database/Migrations'],
],

A migration file is a class extending Migration with up() / down() methods. For schema changes, use CapsuleManager::schema(): 

use Illuminate\Database\Capsule\Manager as CapsuleManager;
use Illuminate\Database\Schema\Blueprint;

CapsuleManager::schema()->create('pages', function (Blueprint $table) {
    $table->increments('id');
    $table->string('slug', 255)->unique();
    $table->timestamps();
});

The history of executed migrations is stored in the migrations table (name from config). Laravel documentation: Migrations.

Seeders

Seeder classes are listed in config/seeders.php and in the component seeders.list. Component seeders implement SeederInterface (the run(): void method); additional application seeders may extend Illuminate\Database\Seeder. 

// config/seeders.php
use Database\Seeders\PageSeeder;

return [
    'seeders' => [
        'list' => [PageSeeder::class],
    ],
];

db:seed with no options runs all registered seeders. To run a single seeder, pass the class full FQCN via -c (or --class) — short class names are not supported. Quote the value because of backslashes in the namespace: 

php bin/console.php db:seed -c "Concept\Components\AuthAdmin\Database\Seeders\UserSeeder"

List available classes: php bin/console.php db:seeders:list.

Console

CommandPurpose
db:migrateApply new migrations
db:rollbackRoll back the last batch
db:migration:listList migrations and status
db:seedAll seeders; -c "Fully\Qualified\Class" — one seeder (full FQCN)
db:seeders:listRegistered seeders
 
php bin/console.php db:migrate
php bin/console.php db:seed
php bin/console.php db:seed -c "Database\Seeders\PageSeeder"
php bin/console.php db:seed -c "Concept\Components\AuthAdmin\Database\Seeders\UserSeeder"

Before the first db:migrate, make sure the database exists and DB_* in .env match your environment.

File layout

config/db.php
config/migrations.php
config/seeders.php
database/migrations/          # application migrations
database/seeders/             # application seeders (namespace Database\Seeders)
src/App/Models/BaseModel.php
src/Components/*/Models/
src/Components/*/Database/Migrations/
src/Components/*/Database/Seeders/

Query API, relationships, scopes, soft deletes, and enum casts work like Laravel Eloquent; the differences are only in bootstrap (Capsule instead of full Laravel) and registering migration/seeder paths through config and components.
