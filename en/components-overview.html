Component Concept

A component in Concept Core is an isolated application feature module (modular monolith): its own routes, templates, migrations, CLI commands, and optional DI providers. The core assembles active components via ComponentRegistry and ComponentsServiceProvider.

See also: directory structure, component providers, creating a component (AuthAdmin).

Why components

  • Package by feature — admin, cabinet, and catalog code live together.
  • Enable/disable — one line in config/components.php.
  • Team boundaries — changes under AuthAdmin/ rarely touch UserCabinet/.
  • Shared core — HTTP, validation, Twig, and DB stay in Concept Core and App/.

ComponentInterface contract

The component class implements Concept\Core\Components\Component\Contracts\ComponentInterface (for example via BaseComponent):

MethodPurpose
name(), version(), description()Metadata for component:list
routes()Absolute path to routes.php or null
providers()Additional module service providers
viewPaths()Twig namespace → path (auth-admin → /src/Components/.../Views)
viewContexts()URI prefix → context (/admin → dashboard)
viewExtensions()Twig\Extension\ExtensionInterface classes
commands()Symfony Console commands
seeders()Seeder classes
migrations()Migration directory paths
assets()source → public map for component:publish-assets

Declare values in the module’s config.php; BaseComponent reads it via Noodlehaus Config.

Application registration

// config/components.php
return [
    'components' => [
        AuthAdminComponent::class => AuthAdminComponent::class,
        UserCabinetComponent::class => UserCabinetComponent::class,
    ],
];

Both key and value are the component class FQCN (the container creates the instance via DI). In config/dev/components.php you can add dev-only modules (for example DebugBar).

Class names are stored in ComponentRegistry when ComponentsServiceProvider is registered (last in bootstrap/providers/app.php).

What happens on boot

ComponentsServiceProvider::boot() for each active component:

  1. Dispatches the ComponentRegistering event
  2. Registers console commands, migration paths, and seeders
  3. Boots nested service providers from config.php
  4. requires routes.php files ($router is available)
  5. In HTTP (not CLI) — adds Twig paths, extensions, and view contexts

Order: core providers → ComponentsServiceProvider → module routes and views.

Moving existing code into a component

Typical refactor from a flat App/ layout:

  1. Create src/Components/MyFeature/ with subdirectories following AuthAdmin.
  2. Move controllers, models, FormRequest, views, and middleware into the component; update namespaces to Concept\Components\MyFeature\....
  3. Move feature routes from routes/web.php to MyFeature/routes.php.
  4. Add MyFeatureComponent.php (extends BaseComponent) and config.php.
  5. Register the class in config/components.php.
  6. Keep only shared code in App/: BaseModel, global layouts, utilities.
  7. Run php bin/console.php db:migrate and component:publish-assets when needed.

If the module uses a shared layout (@dashboard), leave templates in resources/views/dashboard/ — the component only extends them in Twig.

Component vs App code

You do not have to move everything into components. Keep in App/:

  • landing pages, static pages, shared API gateway;
  • base model classes, interfaces, cross-module services;
  • global Twig namespaces (frontend, dashboard) in config/view.php.

Use a separate component when there is a clear boundary: “admin panel”, “user cabinet”, “project catalog”.

One component — one responsibility. If two modules import each other directly, move shared code to App/ or a separate application “core” package.
