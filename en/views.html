Views

HTML in Concept Core is rendered through Twig. Templates are .twig files in the views directory; in the controller the response is built via ViewResponseFactoryInterface::create() (see Responses → HTML).

Low-level rendering — ViewInterface::render() (services, error handler). View paths are configured in config/view.php and in component config.php files.

Twig namespaces

A template from the controller is specified as @namespace/path/without_extension. The namespace is registered in configuration: 

// config/view.php
'paths' => [
    'frontend'  => '/resources/views/frontend',
    'dashboard' => '/resources/views/dashboard',
],

// src/Components/UserCabinet/config.php
'paths' => [
    'cabinet' => '/src/Components/UserCabinet/Views',
],

File src/Components/UserCabinet/Views/index.twig → @cabinet/index.

The resources/views directory (PathManager::VIEWS_DIR key) is also added as a fallback without a namespace — rarely needed if all templates are in namespaced paths.

Rendering from the controller

return $this->viewResponse->create('@frontend/index');

return $this->viewResponse->create('@cabinet/index', [
    'user' => $user,
    'latestProject' => $latestProject,
]);

The second argument — variables for this request only. They are merged with the global context (errors, old, flashes, csrf_token) from ShareViewDataMiddleware.

Layouts and partials

Typical structure:

resources/views/frontend/
├── layouts/base.twig
├── partials/_navbar.twig
├── partials/_flash.twig
└── index.twig

src/Components/UserCabinet/Views/
├── layouts/cabinet.twig
├── layouts/auth.twig
└── index.twig
{# index.twig #}
{% extends '@frontend/layouts/base.twig' %}

{% block title %}Головна{% endblock %}

{% block body %}
    <h1>Контент</h1>
{% endblock %}
{# Підключення фрагмента #}
{% include '@frontend/partials/_flash.twig' %}

Component pages often extend a shared layout from another namespace (for example admin: @auth-admin/users/list.twig → @dashboard/layouts/base.twig).

URLs in templates

UrlExtension extension (config/view.php):

<a href="{{ path('architecture') }}">Architecture</a>

<a href="{{ path('admin.user.edit', { id: user.id }) }}">Edit</a>

  • path() — relative path by route name
  • route() — full URL
  • base_url() — application base

route_namespace

route_namespace is not the same as a Twig template namespace (@cabinet/index). It is the logical UI context: based on the current URL, it determines which «skin» the page uses (public site, admin panel, user cabinet).

The variable is available in all templates as a global: it is added by ContextExtension via ViewContextResolver.

Why

A single application has multiple zones with different layouts and styles. On 404/500 or in shared partials, you need to know which set of views to include, without checking the path in every Twig file.

route_namespace answers: URI prefix → context name from configuration.

How it is determined

  1. In view.contexts (application + components) a map of URI prefix → context is defined.
  2. ViewContextResolver takes the request path, iterates prefixes from longest (so /admin/users matches /admin, not /).
  3. ContextExtension::getGlobals() returns route_namespace to Twig.
// config/view.php
'contexts' => [
    '/' => 'frontend',
],

// src/Components/AuthAdmin/config.php
'contexts' => [
    '/admin' => 'dashboard',
],

// src/Components/UserCabinet/config.php
'contexts' => [
    '/cabinet' => 'cabinet',
    '/login' => 'cabinet',
],
URLroute_namespace
/, /architecturefrontend
/admin/usersdashboard
/cabinet/profilecabinet

Example: error pages

You can use a single dispatcher in resources/views/errors/ that includes 404/500 for the appropriate context: 

{# resources/views/errors/404.twig #}
{% include '@' ~ (route_namespace|default('frontend')) ~ '/errors/404.twig' %}
  • /unknown-page → frontend → @frontend/errors/404.twig (public layout)
  • /admin/missing → dashboard → @dashboard/errors/404.twig (admin layout)

Public 404 extends @frontend/layouts/base.twig; admin — @dashboard/layouts/error.twig. Without route_namespace you would have to duplicate conditions «if /admin — one template, else another».

Other usage in Twig

{# Різний partial залежно від зони #}
{% if route_namespace == 'dashboard' %}
    {% include '@dashboard/partials/_sidebar.twig' %}
{% else %}
    {% include '@frontend/partials/_navbar.twig' %}
{% endif %}

{# Динамічний include (ignore missing — якщо partial відсутній) #}
{% include '@' ~ route_namespace ~ '/partials/_breadcrumbs.twig' ignore missing %}

Important nuance

Values in contexts (dashboard, frontend, cabinet) must match the Twig namespace where the required templates actually live.

In AuthAdmin the context is named dashboard, not auth-admin: error pages and the shared admin layout are in @dashboard/, while CRUD content is in @auth-admin/.

For /cabinet the context is cabinet, but there are no separate Views/errors/ in the UserCabinet component. If you need a custom 404 in the cabinet — add @cabinet/errors/404.twig or change the dispatcher with a fallback to frontend.

config/view.php configuration

return [
    'view' => [
        'extensions' => [
            AppExtension::class,      // has_component()
            ContextExtension::class,  // route_namespace
            UrlExtension::class,      // path(), route(), base_url()
        ],
        'paths' => [ /* namespace => шлях від кореня проєкту */ ],
        'contexts' => [
            '/' => 'frontend',       // URI prefix => ім'я контексту для route_namespace
        ],
        'cache_dir' => 'views',      // storage/cache/views у production
    ],
];
  • extensions — Twig\Extension\AbstractExtension classes, resolved from the DI container
  • paths — Twig namespaces for the application
  • contexts — URI prefix → route_namespace (details)
  • cache_dir — subdirectory in storage/cache; when app.debug Twig cache is disabled

Views in components

In Component/config.php the view section: paths, contexts, optionally extensions. During ComponentsServiceProvider::boot() the data is added to registries and picked up by Twig on HTTP requests (not in CLI). 

// AuthAdmin — власне розширення + шаблони
'view' => [
    'extensions' => [ TwigExtension::class ],  // auth()
    'paths' => [
        'auth-admin' => '/src/Components/AuthAdmin/Views',
    ],
    'contexts' => [
        '/admin' => 'dashboard',
    ],
],

In Twig: {{ auth().check() }} (if the component extension is enabled), {{ has_component('UserCabinet') }} — from AppExtension.

Custom Twig extension

class MyExtension extends AbstractExtension
{
    public function getFunctions(): array
    {
        return [
            new TwigFunction('format_money', [$this, 'formatMoney']),
        ];
    }

    public function getFilters(): array
    {
        return [
            new TwigFilter('excerpt', [$this, 'excerpt']),
        ];
    }
}

Register the class in view.extensions (application) or component view.extensions. In templates: {{ price|format_money }}, {{ format_money(price) }}.

Direct ViewInterface call

/** @var ViewInterface $view */
$view = $container->get(ViewInterface::class);
$html = $view->render('@frontend/errors/404', ['exception' => $e]);

The .twig extension is added automatically if not specified. Events: TemplateRendering, TemplateRendered (telemetry). In debug — DebugExtension and Twig Profiler.

Error pages

The dispatcher resources/views/errors/{code}.twig via route_namespace includes @frontend/errors/... or @dashboard/errors/.... In production the error handler renders the same path; if the file is missing — fallback to 500.

Additional PHP fallback files — resources/views/errors/fallback/ (PathManager::ERRORS_FALLBACK_VIEWS_DIR).

Static assets in templates

CSS/JS usually live in public/ and are linked with absolute paths (/assets/css/frontend.css, /vendor/bootstrap/...). Components can publish assets via assets in config.php (copied to public/components/...).

Keep logic (DB queries, access control) in controllers and services. In Twig — display, conditions, loops, and helper filters.
