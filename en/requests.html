Requests & Validation

Incoming HTTP data (query, form body, JSON) is validated through FormRequest classes. After a successful check, you work with the validated() array or, optionally, a typed DTO built by the Valinor library.

The validator is Rakit Validation (adapter in Concept Core). See also: controllers → method arguments, HandleValidationExceptionMiddleware.

How it works

  1. The controller action declares the type MyRequest $request (a subclass of FormRequest).
  2. RouteStrategy creates the FormRequest from the container and calls validate() before entering the method.
  3. If rules fail — ValidationException → middleware returns 422 (JSON) or redirects back with flash.
  4. If everything is OK — the controller method runs; data is read via validated() or toDto().

With and without DTO

Both approaches are supported. A DTO is not required.

Approach When it is convenient In the controller
With DTO Typed contract, passing to a service $dto = $request->toDto();
Without DTO Direct insert / update in the DB with the validated() array ->create($request->validated())

Example with DTO

class CreateProjectRequest extends FormRequest
{
    protected ?string $dtoClass = CreateProjectDto::class;

    public function rules(): array { /* ... */ }
}

// Controller
/** @var CreateProjectDto $data */
$data = $request->toDto();
$this->projectModel->newQuery()->create([
    ProjectModel::FIELD_TITLE => $data->title,
    // ...
]);

Example without DTO

UpdateProjectStatusRequest — without $dtoClass. The validated() array is passed directly to create() / update() (via a normalizer if needed): 

class UpdateProjectStatusRequest extends FormRequest
{
    public function rules(): array
    {
        return [
            'status' => ['required', 'in:' . implode(',', array_column(ProjectStatus::cases(), 'value'))],
        ];
    }
}

// Single key — update in DB
$project->update([
    ProjectModel::FIELD_STATUS => $request->validated()['status'],
]);

// Multiple fields — create() with the full array (after except / rules)
$this->postModel->newQuery()->create(
    $this->normalizePayload($request->validated())
);

In the controller you can call validated() even if $dtoClass is set in the FormRequest — the DTO is then simply optional (toDto() is not used).

Creating a FormRequest

Place the class in the component or application Requests/ directory (for example Components/UserCabinet/Requests/LoginRequest.php). Extend Concept\Core\Http\Requests\FormRequest and implement rules(). No container registration is needed — autowiring handles it. 

<?php declare(strict_types=1);

namespace Concept\Components\UserCabinet\Requests;

use Concept\Components\AuthAdmin\Dto\LoginDto;
use Concept\Core\Http\Requests\FormRequest;

/** @extends FormRequest<LoginDto> */
class LoginRequest extends FormRequest
{
    protected ?string $dtoClass = LoginDto::class;

    /** Fields that will not appear in validated() (even if they passed validation) */
    protected array $except = [];

    /** If not empty — only these keys remain in validated() */
    protected array $only = [];

    public function rules(): array
    {
        return [
            'email' => ['required', 'email'],
            'password' => ['required'],
            'remember' => ['nullable', 'boolean'],
        ];
    }

    public function aliases(): array
    {
        return [
            'email' => 'Email Address',
            'password' => 'Password',
        ];
    }
}

rules() rules

Two formats (as in Rakit):

  • string: 'name' => 'required|min:3'
  • array: 'email' => ['required', 'email', 'unique:users,email']

Standard Rakit rules are available (required, email, min, max, same, in, nullable, boolean, etc.). You can create custom rules: unique, exists (see custom rules).

aliases() aliases

Human-readable field names in error messages («Email Address» instead of «email»). Optional method — defaults to an empty array.

$except and $only

  • $except — exclude fields from the validated() result (for example password_confirmation after the same:password rule)
  • $only — whitelist of keys in validated() (rare; usually the keys from rules() are enough)
  • _csrf_token is excluded globally from validated(), even if it appears in the body

validated() returns only keys present in rules() (intersection with valid data), minus except and global exclusions. You can call validated() only after validate() (in the controller validation has already run; otherwise — ValidationLogicException).

Where data comes from: all()

all() merges the query string and parsed body (form fields or JSON after ParseJsonBodyMiddleware). This data is passed to the validator. For JSON APIs, attach the appropriate middleware to the route group.

$dtoClass property

  • protected ?string $dtoClass = MyDto::class; — enable DTO mapping
  • null or absent — only validated() works; toDto() returns null

PHPDoc @extends FormRequest<LoginDto> helps IDE/static analysis; for requests without a DTO — @extends FormRequest<null> or no generic.

Route parameters

URL segments ({id:number}) are usually passed as a separate action argument: update(UpdateUserRequest $request, int $id). In a FormRequest subclass, the protected getRouteParam('id') is available — if you need to inject id into rules (for example unique:users,email,{id}).

Logging

When log.validation_data = true in configuration, FormRequest logs input data and validation results (see Configuration). Usually disabled in production.

Usage in the controller

Declare FormRequest in the method signature — validation runs automatically:

public function store(StoreUserRequest $request): ResponseInterface
{
    // Option 1: array
    $this->userModel->newQuery()->create(
        $this->normalizeUserPayload($request->validated())
    );

    return $this->response->redirectByName('admin.users');
}

public function login(LoginRequest $request): ResponseInterface
{
    // Option 2: DTO
    $loginDto = $request->toDto();
    if ($loginDto) {
        $this->auth->attempt($loginDto->email, $loginDto->password, $loginDto->remember);
    }
    // ...
}

Call toDto() after successful validation (in the controller method this is already guaranteed). Check for null if $dtoClass is not set or Caster is unavailable.

Validation errors

On failed validation:

  • JSON / API — 422, body via jsonError() with an array of errors
  • HTML form — flash error, _validation_errors, _validation_data, redirect back()

ShareViewDataMiddleware passes Twig variables errors and old (previous input): 

<label for="email">Email</label>
<input id="email" type="email" name="email" required
       value="{{ old.email|default('') }}"
       class="input {{ errors.email is defined and errors.email ? 'is-invalid' : '' }}"
       aria-describedby="email-error">
{% if errors.email is defined and errors.email is not empty %}
    <div class="field-error" id="email-error" role="alert">
        {{ errors.email is iterable ? errors.email|join(', ') : errors.email }}
    </div>
{% endif %}

DTO mapping errors

If toDto() cannot build the object (type mismatch after validation), ValidationCastException is thrown (wrapper over Valinor MappingError). This is not the same as rule errors — handled by the general error handler, not validation middleware. Therefore FormRequest rules must align with DTO types (for example boolean for bool $remember).

Custom validation rules

Registration in config/validator.php:

return [
    'validator' => [
        'rules' => [
            'unique' => UniqueRule::class,
            'exists' => ExistsRule::class,
        ],
    ],
];

The rule class implements RuleInterface, extends Concept\Core\Components\Validator\Rule. In rules: 'email' => ['unique:users,email'], 'email' => ['unique:users,email,5,id'] (except id on update).

DTO (Data Transfer Object)

A DTO is an immutable object with readonly properties for already validated data. The base class Concept\Core\Dto\Dto implements toArray() via casting the object to an array.

How to create a DTO

<?php declare(strict_types=1);

namespace Concept\Components\AuthAdmin\Dto\User;

use Concept\Core\Dto\Contracts\DtoInterface;
use Concept\Core\Dto\Dto;

class StoreUserDto extends Dto implements DtoInterface
{
    public function __construct(
        public readonly string $name,
        public readonly string $email,
        public readonly string $password,
        public readonly string $status,
        public readonly bool $is_admin = false,
        public readonly ?string $verification_token = null,
    ) {}
}
  • Property names match keys in validated() (snake_case in forms → snake_case in DTO)
  • Constructor types must match what the validator + Valinor casting returns
  • Optional fields — default in the constructor or ?type
  • Fields in $except will not appear in the DTO

Valinor and Caster

Mapping is performed by Concept\Core\Components\Caster\Caster based on Valinor (cuyz/valinor in core dependencies; an internal fork with the same CuyZ\Valinor API is possible — then the same documentation applies). It describes mapping arrays to objects, strict types, nested structures — applicable to Concept Core DTOs.

  • allowScalarValueCasting() — string "1" → int / bool when needed
  • allowSuperfluousKeys() — extra keys in the array do not break mapping (convenient for forms)
  • Mapper cache: storage/cache/valinor (faster repeated mapping)

Useful Valinor documentation sections:

  • How to map input data
  • Handle invalid input
  • Feature flags (casting, superfluous keys)

What toDto() does

// Simplified
if ($this->dtoClass === null || !$this->caster) {
    return null;
}
return $this->caster->cast($this->validated(), $this->dtoClass);

First Rakit checks rules, then Valinor builds the structure. Business-logic validation (email uniqueness in DB) — in rules(); type coercion — in DTO/Valinor.

Checklist

  1. Create Requests/MyActionRequest.php with rules() and aliases() if needed.
  2. If needed — Dto/MyActionDto.php and protected ?string $dtoClass = ....
  3. Add a type hint in the controller: public function store(MyActionRequest $request).
  4. Ensure global HandleValidationExceptionMiddleware is attached to routes.
  5. In Twig use errors / old to re-display the form.

FormRequest is responsible for request input data. Authorization («can this user») belongs in middleware or services, not in rules(), if session context beyond form fields is required.
