Data Masking

The core redacts sensitive values before they are written to logs, debug panels, and other diagnostics. The Masker component recursively processes arrays, objects, and strings using key patterns and regular expressions.

Registration: MaskerServiceProvider in HTTP providers (before LogServiceProvider so the logger receives the masker).

Architecture

ClassRole
MaskerInterfaceContract: mask(mixed $data): mixed
DataMaskerRule orchestrator, recursive traversal
RegexMaskerBuilt-in string and key patterns
MaskingRuleInterfaceCustom rule (apply, isSensitiveKey)

By default MaskerServiceProvider builds DataMasker with a single RegexMasker rule. The service is registered as a shared singleton.

Built-in rules

Keys (case-insensitive patterns) are replaced with ***:

  • names containing password, token, _csrf_token, secret
  • api_key, authorization

Strings are processed with regex:

  • email → ***@***.***
  • card number 9999-9999-9999-9999 → asterisks
  • pairs like password=..., token=..., api_key=... → value replaced with *****

Logger integration

LogServiceProvider passes MaskerInterface into Concept\Core\Components\Logger\Logger. A Monolog processor masks $record->context before writing to file or stderr. 

$logger->info('User updated', [
    'email' => 'user@example.com',
    'password' => 'secret123',  // logged as ***
]);

Log messages themselves are not masked — keep secrets in context only.

Manual usage

use Concept\Core\Components\Masker\Contracts\MaskerInterface;

/** @var MaskerInterface $masker */
$masker = $container->get(MaskerInterface::class);

$safe = $masker->mask([
    'login' => 'oleh',
    'password' => 'hunter2',
    'card' => '4111-1111-1111-1111',
]);

For arrays and objects, mask() uses deepClone first so the original is unchanged. maskRecursive() mutates the structure in place when cloning is unnecessary.

Custom rules

In your own provider (after MaskerServiceProvider), you can override the binding:

use Concept\Core\Components\Masker\Contracts\MaskerInterface;
use Concept\Core\Components\Masker\DataMasker;
use Concept\Core\Components\Masker\RegexMasker;

$masker = new DataMasker();
$masker->addRule(new RegexMasker(
    customPatterns: ['/iban=\d+/i' => 'iban=***'],
    customKeyPatterns: ['/.*_pin$/i']
));

$container->add(MaskerInterface::class, fn () => $masker)->setShared(true);

Or implement MaskingRuleInterface for more complex logic (for example partial phone masking).

DebugBar and other tools

The DebugBar component injects the masker into its request collector so $_POST and $_SESSION are shown redacted. Follow the same pattern in custom diagnostics.

Masking reduces leakage via logs and dev tools; it does not encrypt data at rest or strip fields from HTTP responses.
