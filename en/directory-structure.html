Directory Structure

Concept Core does not impose any directory structure. The location of all folders — including src, config, storage, database — can be customized in bootstrap/paths.php and in the PSR-4 autoloading section of composer.json.

Below is the recommended root structure for a project and two common approaches to organizing code inside the src folder. These are guidelines, not a mandatory standard.

Root directories (public, bootstrap, config, etc.) — a recommended convention to get started. They can be renamed or moved if you update paths.php and the web server configuration.

Recommended root structure

project/
├── public/           # document root, index.php
├── bootstrap/        # app.php, paths.php, providers/
├── bin/              # console.php
├── config/           # PHP config + config/{env}/
├── routes/           # routes (loaded from config)
├── resources/views/  # shared Twig templates (layouts)
├── database/         # default migrations/seeders
├── storage/          # logs, cache
├── src/              # application code (organization — variants below)
└── tests/

The path to each of these directories is set in paths.php (CONFIG_DIR, SRC_DIR, LOGS_DIR…). For example, you can keep code in app/ instead of src/ — just change PathManager::SRC_DIR and the Composer namespace mapping.

Organizing code in the src directory

Two typical variants follow. Both are compatible with Concept Core.

1. Layered MVC (package by layer)

Classic layered architecture: code is grouped by type (controllers, models, requests), not by module. In the PHP community this is often called package by layer — the opposite of "package by feature".

Layers sit directly in the code directory (e.g. src/), without intermediate folders like App or Components: 

src/
├── Controllers/       # Concept\App\Controllers\*
├── Models/            # Concept\App\Models\*
├── Requests/          # FormRequest, DTO requests
├── Validation/
│   └── Rules/
├── Extensions/
│   └── Twig/
├── Services/          # optional: business logic
└── Middlewares/       # optional: HTTP middleware

When to choose: small applications, single domain, few modules; team is used to classic MVC (Symfony 2–4, early Laravel).

Drawbacks: as the project grows, it is hard to find "everything about orders" — files are spread across layers.

Composer (example):

"autoload": {
    "psr-4": {
        "Concept\\": "src/"
    }
}

paths.php: PathManager::SRC_DIR => 'src' (details).

2. Components (modular monolith)

A modular monolith with pluggable components: shared application core + isolated functional modules. This combines package by feature (inside each component) with MVC layers. Supported by the Concept Core component system. 

src/
├── App/                          # shared application code
│   ├── Controllers/              # global controllers
│   ├── Models/                   # BaseModel, shared traits
│   ├── Requests/
│   ├── Validation/Rules/
│   └── Extensions/Twig/
└── Components/                   # functional modules
    ├── AuthAdmin/
    │   ├── Controllers/
    │   ├── Models/
    │   ├── Views/                # module Twig templates
    │   ├── routes.php
    │   ├── config.php
    │   ├── Database/Migrations/
    │   └── AuthAdminComponent.php
    ├── UserCabinet/
    └── ProjectManager/

Components are registered in config/components.php, and can have their own routes, views, migrations, and assets. The App directory holds shared pieces (base models, Twig extensions, validation rules).

When to choose: multiple subsystems (admin, user cabinet, public site); need to enable or disable modules via config; different teams work on separate components.

Comparison

Layered MVC Components
Academic term Layered / package by layer Modular monolith, package by feature
Project size Small — medium Medium — large
Routes routes/web.php + routes.php in each component
Concept Core Full support + component subsystem

Where to store views and data

Paths are configured again in paths.php.

  • Shared layouts — typically resources/views/ (VIEWS_DIR).
  • Module templates — in the component variant: a Views/ directory inside the module.
  • Migrations — typically database/migrations; additional paths in config/migrations.php.
  • Logs, cache — typically storage/ (LOGS_DIR, CACHE_DIR).

What to choose

  • Layered MVC — simple start, single domain, few modules.
  • Components — multiple subsystems, module isolation, Concept Core component support.

You can change the structure at any time: update paths.php, composer.json, and config.
