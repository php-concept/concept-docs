Service Providers

A service provider is a class that registers services in the DI container during bootstrap. In Concept Core, this is the primary way to wire up config, HTTP, database, Twig, validation, and component modules.

The implementation follows League Container Service Providers (League\Container\ServiceProvider\AbstractServiceProvider). See also: Service Container, configuration.

Provider anatomy

<?php declare(strict_types=1);

namespace Concept\App\Providers;

use Concept\Core\Providers\Concerns\PeeksEventDispatcher;
use League\Container\ServiceProvider\AbstractServiceProvider;
use League\Container\ServiceProvider\BootableServiceProviderInterface;

final class BillingServiceProvider extends AbstractServiceProvider implements BootableServiceProviderInterface
{
    use PeeksEventDispatcher;

    public function provides(string $id): bool
    {
        return $id === BillingGatewayInterface::class;
    }

    public function register(): void
    {
        $container = $this->getContainer();

        $container->add(BillingGatewayInterface::class, function () {
            return new StripeBillingGateway(/* ... */);
        })->setShared(true);
    }

    public function boot(): void
    {
        // Eager logic immediately after register() of this provider
    }
}

register() and boot()

  • register() — $container->add(), bindings, factories
  • boot() — additional logic immediately after register() of this same provider (eager), not after all providers in the file

boot() only makes sense in a class that implements BootableServiceProviderInterface. From the League Container documentation:

In much the same way, this method has access to the container itself and can interact with it however you wish, the difference is that the boot method is invoked as soon as you register the service provider with the container meaning that everything in this method is eagerly loaded.

If you wish to apply inflectors or register further service providers from this one, it must be from a bootable service provider like this one, otherwise they will be ignored.

League Container — Service Providers

Order in bootstrap/providers/app.php is critical: for each provider, register() runs first, then boot() immediately, and only then the next provider in the list. In provider N's boot(), only services from providers above in the file are available; providers below have not yet run register().

That is why ComponentsServiceProvider is last: its boot() already has ConfigInterface, Router, View registries, and more — and it can addServiceProvider() for component providers, load routes.php, and Twig paths.

provides()

A list of ids this provider supplies. Enables deferred loading: the provider is loaded when one of the listed services is first needed.

ServiceAwakening events

Core providers use the PeeksEventDispatcher trait: if EventDispatcherInterface is already in the container, creating a service dispatches ServiceAwakening (telemetry in debug mode).

Registering providers

App::registerServiceProviders() loads an array of classes from a PHP file:

foreach ($providers as $providerClassName) {
    $provider = new $providerClassName();
    $this->container->addServiceProvider($provider);
}

Providers are not registered via autowiring — they are instantiated explicitly (new $providerClassName()).

Core providers — purpose

Each class below is part of php-concept/core. Order in the array is critical: a provider only sees what providers above it in the list have registered in the container.

Provider Why needed What it registers
ConfigServiceProvider First in the list: gives access to application configuration. Loads PHP files from config/, variables from .env (phpdotenv), overrides from config/{APP_ENV}/, and sets the timezone. Without it, other providers cannot read db.*, log.*, session.*, and so on. ConfigInterface
ErrorHandlerServiceProvider After config, reconfigures Whoops: instead of the "early" handler from App::create(), it wires the full stack — PrettyPage (debug), fallback templates (production), JSON for API, plain text for CLI. Also writes unhandled exceptions to the log via ErrorLogHandler. Whoops (handlers in boot())
EventServiceProvider Central event dispatcher for the core (League Event). In app.debug mode, collects telemetry (ServiceAwakening, SQL queries, and so on) into a buffer for the debug panel and bootstrap diagnostics. EventDispatcherInterface, ApplicationTelemetryBuffer
HttpServiceProvider HTTP layer of the web application: builds a PSR-7 request from superglobals, configures League Route with RouteStrategy (controller autowiring and FormRequest), loads route files from config/routes.php, and provides response factories and URL generation. In boot() — page resolvers for Illuminate Paginator. ServerRequestInterface, Router, UrlGeneratorInterface, ResponseFactoryInterface, ViewResponseFactoryInterface, RequestFormat
SessionServiceProvider State between HTTP requests: authentication, flash messages, data after failed validation (errors, old in Twig), CSRF token. Cookie options come from config/session.php. SessionInterface, FlashBagInterface
MaskerServiceProvider Masks passwords, tokens, and other sensitive values before writing to the log (regex rules). Must be registered before LogServiceProvider so Monolog receives the masker processor immediately. MaskerInterface
LogServiceProvider Primary application logging via Monolog: rotation of storage/logs/app.log, level from log.level, channel name from log.name. Wraps Monolog in a Logger with masker integration. LoggerInterface
ViewServiceProvider HTML rendering via Twig: base path resources/views, cache (disabled in debug), extensions and namespaces from config/registries. Components add their paths later via ComponentsServiceProvider. ViewInterface, ViewPathRegistry, ViewExtensionRegistry, ViewContextRegistry
ValidationServiceProvider Input validation for FormRequest: adapter over Rakit Validation plus custom rules from config/validator.php (for example unique, exists for the database). ValidatorInterface, Rakit\Validation\Validator
DatabaseServiceProvider Database connection via Eloquent Capsule: query builder, models, migrations, and seeders. In boot(), creates and configures CapsuleManager from db.* config; in debug or when log.query is enabled, logs SQL. Requires ConfigInterface and (for SQL logging) LoggerInterface. DatabaseInterface, CapsuleManager, Migrator, SeederManager, MigrationRegistry, SeederRegistry
CastingServiceProvider Maps arrays to typed DTOs via Valinor (FormRequest::toDto()). Mapper cache — storage/cache/valinor. Required when FormRequest declares $dtoClass. CasterInterface
DebugLoggerServiceProvider Lightweight debug logging to storage/logs/debug.log via static DebugLogger::log() — handy for quick diagnostics during bootstrap or in places where full Monolog is not yet available. In boot(), sets the singleton instance. DebugLogger
ComponentsServiceProvider Last in the list: loads active modules from config/components.php. In boot(), registers routes, Twig paths/extensions, CLI commands, migration/seeder paths, and nested service providers for components. Requires Router, View registries, Console, and so on — so always after the base providers. ComponentRegistry; side effects in boot()

Order matters: config → events → HTTP/session/log/view/validation/DB → components last.

CLI — bootstrap/providers/console.php

bin/console.php calls App::create() and a separate provider list (without View/Validation/Casting as needed, but with ConsoleServiceProvider): 

// bin/console.php
$app->registerServiceProviders([$rootPath . '/bootstrap/providers/console.php']);

$console = $container->get(ConsoleApplication::class);
$console->run();

ConsoleServiceProvider is the only additional core provider for CLI (details in the provider table):

Provider Why needed What it registers
ConsoleServiceProvider Entry point for bin/console.php: creates a Symfony Console Application with name/version from config, loads commands from config/commands.php. If a command cannot be resolved from the container — registers DisabledCommand with an error message. Component commands are added by ComponentsServiceProvider::boot(). ConsoleApplication

Application provider

  1. Create a class in App/Providers/MyServiceProvider.php.
  2. In register() — bindings for application interfaces.
  3. Add the class to bootstrap/providers/app.php after the providers it depends on.
// bootstrap/providers/app.php
return [
    ConfigServiceProvider::class,
    // ...
    BillingServiceProvider::class,  // after ConfigServiceProvider
    ComponentsServiceProvider::class,
];

If the provider needs ConfigInterface — place it after ConfigServiceProvider. If it needs Router — after HttpServiceProvider.

Component providers

A component declares providers in Component/config.php. ComponentsServiceProvider::boot() loads them dynamically: 

// src/Components/DebugBar/config.php
'providers' => [
    DebugBarServiceProvider::class,
    MaskerServiceProvider::class,  // if not already in app.php
],
// Concept\Core\Providers\ComponentsServiceProvider
foreach ($registry->providers() as $providerClass) {
    $this->getContainer()->addServiceProvider(new $providerClass());
}

The list of active components is in config/components.php (in dev, config/dev/components.php may include DebugBar). The component class (AuthAdminComponent) reads its own config.php.

What ComponentsServiceProvider::boot() does

  1. ComponentRegistering events for each module
  2. Registration of component console commands
  3. Migration and seeder paths in registries
  4. Nested service providers of components
  5. require of Component/routes.php files (the $router variable)
  6. In HTTP (not CLI) — Twig paths, extensions, view contexts from components

So component routes appear only if the component is enabled in config and ComponentsServiceProvider has run boot().

Examples from core

HttpServiceProvider — register + boot

register(): PSR-7, Router + RouteStrategy + loading routes/*.php.

boot(): page resolvers for Illuminate Paginator.

League Route: routing documentation.

DatabaseServiceProvider — why boot

In register() — DatabaseInterface, Migrator, registries. In boot() (immediately after this provider's register()) — creating CapsuleManager, connection from ConfigInterface (Config is already registered higher in the list), SQL listeners. Capsule is added to the container already configured.

ViewServiceProvider — Twig

Builds the Twig Environment: cache, debug, extensions from config, paths from registries. After component boot, additional namespaces appear in the registries.

Twig: twig.symfony.com, views — Views.

Recommendations

  • One provider — one area (Billing, Notifications), not a "GodProvider"
  • register — bindings only; heavy wiring — in boot()
  • Dependencies via $container->get() only inside the provider, not in controllers
  • setShared(true) for services that should be singleton per request
  • Do not change core provider order without understanding dependencies

New module checklist

  1. Component in config/components.php
  2. If needed — providers in Component/config.php
  3. routes.php, view.paths, migrations — in the component config
  4. Service classes resolve via autowiring or binding in the module provider

If a service is "not found" in the container — check whether the provider is in the correct file (app.php vs console.php), whether the component's boot() ran, and whether an explicit add() is needed instead of autowiring.
