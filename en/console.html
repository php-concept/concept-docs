Working with the Console

The application CLI is built on Symfony Console. The entry point is bin/console.php; commands are registered via the DI container and ConsoleServiceProvider.

See also: setup → bin/console.php, CLI providers, creating custom commands.

Running

From the project root:

php bin/console.php

# or via Composer (script in composer.json)
composer console

# help
php bin/console.php list

# help for a single command
php bin/console.php help db:migrate

CLI bootstrap differs from HTTP: it loads bootstrap/providers/console.php (without View/Validation for the web stack, with ConsoleServiceProvider). For details, see Setup.

How it works

// bin/console.php
$app = App::create($rootPath, $paths);
$app->registerServiceProviders([$rootPath . '/bootstrap/providers/console.php']);

$console = $app->getContainer()->get(ConsoleApplication::class);
$console->run();
  1. ConsoleServiceProvider creates Symfony\Component\Console\Application
  2. Commands from config/commands.php are resolved from the container and added to the application
  3. ComponentsServiceProvider::boot() adds commands from each component's config.php

Core commands

Registered in config/commands.php:

CommandPurpose
db:migrateRun migrations
db:migration:listList migrations
db:rollbackRoll back the last migration batch
db:seedAll seeders; -c "Fully\Qualified\Class" — one seeder (full FQCN)
db:seeders:listList registered seeders
route:listRoute table (-F — full middleware names)
view:clearClear Twig cache
component:listList enabled components
component:publish-assetsPublish component assets to public/

Examples

php bin/console.php db:migrate
php bin/console.php db:seed
php bin/console.php db:seed -c "Concept\Components\AuthAdmin\Database\Seeders\UserSeeder"
php bin/console.php route:list
php bin/console.php route:list -F
php bin/console.php component:list
php bin/console.php view:clear

Component commands

A component adds commands in the commands section of its config.php. During ComponentsServiceProvider::boot() they are registered in the same ConsoleApplication.

Exit codes

Commands return Command::SUCCESS (0) or Command::FAILURE (1). In CI scripts, check the exit code after php bin/console.php ....

Environment and config

CLI uses the same .env and ConfigServiceProvider as HTTP. APP_ENV determines config/{env}/ overrides and the component list (for example, DebugBar only in config/dev/components.php).

Migrations and seeders require a valid DB connection in config/db.php (or via .env).

Use the console for maintenance (migrations, cache, diagnostics) and local development. Keep long-running business logic in services; leave a thin call in the command.
