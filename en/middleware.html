Middleware

Middleware is a layer between routing and the controller: it filters the request, adds context, catches exceptions, or modifies the response. Concept Core uses Psr\Http\Server\MiddlewareInterface (PSR-15) and League Route lazyMiddleware().

See also: request lifecycle → middleware, registration on routes.

PSR-15 contract

<?php declare(strict_types=1);

namespace App\Http\Middlewares;

use Psr\Http\Message\ResponseInterface;
use Psr\Http\Message\ServerRequestInterface;
use Psr\Http\Server\MiddlewareInterface;
use Psr\Http\Server\RequestHandlerInterface;

final class ExampleMiddleware implements MiddlewareInterface
{
    public function process(
        ServerRequestInterface $request,
        RequestHandlerInterface $handler
    ): ResponseInterface {
        // Before controller: checks, modifying $request

        $response = $handler->handle($request);

        // After controller: headers, logging

        return $response;
    }
}

The class is registered in the container; League Route creates an instance via DI when calling lazyMiddleware(MyMiddleware::class).

Where to attach

  • Global — at the top of routes/web.php (all routes in the file).
  • Group — $router->group(...)->lazyMiddleware(...).
  • Single route — chain after get() / post().
  • Component — in Component/routes.php (for example DebugBar in dev).

Registration order determines wrapping order: the first middleware added is the outer layer (runs first on the way in).

Core middleware

Classes in Concept\Core\Http\Middlewares:

Class Purpose
HandleValidationExceptionMiddleware Catches ValidationException: JSON 422 or flash + redirect back
VerifyCsrfTokenMiddleware CSRF for POST, PUT, DELETE
StorePreviousUrlMiddleware Stores URL in session for back() / safe redirect
ShareViewDataMiddleware Flash, validation errors, csrf_token in Twig
ParseJsonBodyMiddleware Parses JSON body into getParsedBody() (API)
ForceJsonResponseMiddleware Forces Accept/Content-Type JSON

Typical global stack

In routes/web.php, middleware is often attached in this order:

$router->lazyMiddleware(HandleValidationExceptionMiddleware::class);
$router->lazyMiddleware(VerifyCsrfTokenMiddleware::class);
$router->lazyMiddleware(StorePreviousUrlMiddleware::class);
$router->lazyMiddleware(ShareViewDataMiddleware::class);
  1. Outer validation try/catch
  2. CSRF check before state changes
  3. URL capture before rendering
  4. Template data preparation

CSRF

The token is stored in the session (CsrfTokenManager). In forms — the _csrf_token field; for AJAX — headers X-CSRF-TOKEN or X-XSRF-TOKEN. 

<input type="hidden" name="_csrf_token" value="{{ csrf_token }}">

The csrf_token Twig variable is provided by ShareViewDataMiddleware.

Authorization middleware (example)

In the application — custom classes on route groups:

  • AuthMiddleware — admin area: session, admin role, ACTIVE status
  • ActiveUserMiddleware — user cabinet: authentication, pending/blocked statuses

On failure — redirect to login (often redirectByName('cabinet.login')), without running the controller.

$router->group('/cabinet', function (RouteGroup $route) {
  // ...
})->lazyMiddleware(ActiveUserMiddleware::class);

Request attributes

Middleware can pass data downstream via $request->withAttribute():

  • RequestAttribute::VIEW_PAYLOAD — context for Twig
  • RequestAttribute::SAFE_BACK_URL — safe URL for ResponseFactory::back()

The controller or ViewResponseFactory reads attributes from the PSR-7 request.

Custom middleware

  1. Create a class implementing MiddlewareInterface.
  2. Dependencies — via constructor (the container injects them automatically).
  3. Attach with lazyMiddleware globally, on a group, or on a route.
  4. Check order relative to CSRF, auth, and the validation handler.

Middleware should be "narrow": one responsibility. Complex business logic belongs in a service or controller.

Listing middleware

php bin/console.php route:list
php bin/console.php route:list -F

The middleware column shows global and local classes for each route.
